【59】zookeeper与Redis解决分布式锁问题

2018-12-08

note：自己公司现在的用的dubbo，分布式锁也是用zookeeper实现的，今天总结一下

zookeeper实现分布式锁

zookeeper java客户端我们使用Apache curator，相较于原生zookeeper Java API，有以下优点：

  1. 超时重连
  2. 解决watch注册事件一次就失效问题
  3. 支持递归节点操作
  4. 封装了很多zookeeper工具类

好了，不说废话了，开始主题吧！


上面是zk解决分布式锁的流程图，获取资源前先要判断该资源是否被其他线程占用，判断资源是否被占用的方式是去zk寻找该资源对应的节点是否被创建，如果该资源对应的节点已经被创建，则说明该资源已经被其他线程占用，已经被其他线程加锁。如果被其他线程占用，肯定就需要等待锁的释放，阻塞当前线程进行等待，可以使用CountDownLatch进行阻塞。

那么阻塞的线程如何知道锁已经被释放了呢？当该资源对应的zk的节点被删除时，表示锁已经被释放，这就需要使用的zk的watch机制，对锁对应的节点进行监控，一旦该节点被删除，阻塞的线程立马就会知道。

还有一个问题，如果持有锁的线程发生异常一直不释放锁怎么办？导致大量的线程一直等待，进一步可能会导致整个系统的崩溃。这个问题正好zk也能解决，锁节点使用临时节点。如果持有锁的线程出错，zk通过心跳检测，发现该线程长时间没有应答，session断开，会自动删除锁（临时节点）。

另外一个问题就是关于锁的可重入性。这个问题可以通过给锁节点添加数据来解决，通过判断锁节点的数据从而达到可重入。

Redis实现分布式锁

Redis实现分布式锁的流程图如下：


首先进行setnx原子操作，如果不存在则创建key为lockkey的节点，value为 currentTime + timeOut，然后设置有效期，执行业务，释放锁（删除数据）。

如果setnx操作失败，则获取当前key的value，为lockvalueA：

  1. 如果lockvalueA!=null，且currentTime > lockvalueA ，说明该锁持有已经过期，当前线程有权利重新设置该锁，然后设置getset lockvalue = currentTime + timeOut，返回值为旧的lockvalueB：
    1.如果之前的lockvalue不为null，且值没改变（lockvalueA == lockvalueB），则说明该线程可以获取锁
    2.否则获取锁失败
  2. 否则，获取锁失败

死锁问题解决：

  1. 如果线程A setnx成功，expire操作成功，但是在执行业务操作时发生异常，不会导致死锁问题
  2. 如果线程A setnx成功，但是在设置expire之前发生异常，会导致分布式死锁问题，上图中也给出了解决方案，线程B会去判断上一个线程设置的lockvalue 与 currentTime + timeOut，从而解决死锁问题
