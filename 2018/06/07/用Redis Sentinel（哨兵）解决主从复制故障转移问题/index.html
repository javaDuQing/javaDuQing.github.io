【18】用Redis sentinel（哨兵）解决master节点故障转移问题

2018-06-07

主从复制存在一个问题：如果master节点发生了故障怎么办？

能想象到的做法就是我们手动将一个slave节点设置为新的master节点，改变其他slave的master。但是这一过程较为复杂容易产生错误。Redis sentinel提供了主从复制高可用的方案，能够很好的处理master宕机问题。

2.1 故障转移详细过程

主观下线与客观下线

当某个sentinel哨兵在检测master节点时，发现master已经下线，则此时称为master节点主观下线。此时该哨兵会向其他哨兵发送命令（sentinel is-master-down-by-add），让他们也去确认该master节点是否真的下线了。如果认为master下线的哨兵超过我们设定的值quorum（设定的是2），则认为该master节点客观下线，需要进行故障转移。

一般quorum的数量 = sentinel数量/2 + 1

Sentinel领导选举

当确认过master客观下线后，需要选举一个sentinel节点作为本次故障转移的领导。每个做主观下线的sentinel节点向其他sentinel节点发送sentinel is-master-down-by-add有两个含义：

  1. 咨询其他sentinel节点，master是否下线
  2. 咨询其他sentinel是否同意把他设置为领导者（收到sentinel节点如果没有同意过其他sentinel节点发送的命令，那么将同意该请求，否则拒绝）

如果该sentinel节点发现自己的票数已经超过sentinel集合半数，且超过quorum，那么他将成为领导者。

故障转移

  1. 选举的领导sentinel从slave节点中选出一个“合适”的节点作为新的master节点
  2. 对选择的slave节点执行slave no one 命令让其成为master节点
  3. 想剩余的slave节点发送命令，让他们成为新master节点的slave节点，复制规则和parallel-syncs配置有关
  4. 通知客户端主从变化
  5. 更新对原来master节点配置为slave，并保持对其“关注”，当其恢复后，命令他去复制新的master节点数据

Parrallel-syncs参数：例如一个master，两个slave，该参数用来配置是否同步更新两个slave；还是一个一个更新slave。

3 其他问题

3.1 如果选取一个“合适”的节点作为新的master？

  1. 选择slave-priority（slave节点优先级）最高的slave节点，如果存在则返回，不存在则继续
  2. 选择复制偏移量最大的slave节点（复制的最完整），如果存在则返回，不存在则继续
  3. 选择runid最小的slave节点

每个slave节点都是可以设置优先级的，一般情况下，都是使用默认优先级

3.2 如果我们想要master下线，应该怎么办？

选择某个sentinel节点，执行命令 sentinel failover masterName即可。

3.3 Sentinel下的读写分离？

如果说一个slave/master节点下线了，连接这个slave/master节点的client就会出现问题（例如Jedis）。Sentinel没给这种问题的解决方案。但是他会告诉client，slave/master出现了问题，这就需要我们自己变成去更换节点。

JedisSentinelpool 中有解决方案：

  1. 新建一个线程去订阅sentinel消息
  2. 当得知slave/master下线后，重新初始化jedispool，将下线的slave/master剔除

这可以在Jedis的源码中看到。
