【39】Java7 读写锁 ReentrantReadWriteLock 源码分析

2018-08-09

1 写在前面

对于共享资源，ReentrantReadWriteLock允许多个读线程同时访问，但不允许写线程和读线程、写线程和写线程同时访问。但是持有写锁的线程，可以对资源加读锁（锁降级）；在实际应用中，大部分情况下对共享资源（如缓存）的访问都是读操作远多于写操作，这时ReentrantReadWriteLock能够提供比排他锁更好的并发性和吞吐量。

下面的代码展示了如何利用重入来执行升级缓存后的锁降级。

在processCachedData方法中，首先增加读锁。如果缓存中无数据，则先释放读锁，增加写锁（注意，在释放读锁，增加写锁的这个过程中，有可能别的线程获取了锁并在缓存中增加了数据）。 然后再次判断缓存中是否有数据，如果没有则添加数据。在释放写锁之前通过获取读锁来降级，这样做的目的是为了防止释放写锁后被其他线程增加写锁，保证一定能获取读锁。

2 源码分析

2.1 读锁获取过程

那就从rwl.readLock().lock()这句代码开始吧！

先来看代码中的注释：

  1. 如果有其他线程持有写锁，则获取读锁失败
  2. 判断当前线程是否应该阻塞，如果不需要阻塞，则尝试CAS更改state和更改当前线程的重入数。注意：为了避免在更典型的非重入情况下检查重入数量，此过程没有处理锁重入问题，推迟到fullTryAcquireShared方法中处理。（一般情况下，很少会用到重入锁，我是这样理解的）
  3. 如果CAS失败，线程符合条件或者持有读锁的线程数量饱和，则转到fullTryAcquireShared方法中自旋尝试获取读锁。

在tryAcquireShared方法中，有一些变量需要着重解释下：

  1. state 的高 16 位代表读锁的获取次数，包括重入次数，获取到读锁一次加 1，释放掉读锁一次减 1
  2. state 的低 16 位代表写锁的获取次数，因为写锁是独占锁，同时只能被一个线程获得，所以它代表重入次数
  3. cachedHoldCounter 用于记录最后一次获取读锁的线程的HoldCounter。因为在ReentrantReadWriteLock 源码中有一个ThreadLocal变量记录线程的HoldCounter，用cachedHoldCounter 保存比从ThreadLocal变量中获取效率更高
  4. firstReader 和 firstReaderHoldCount，用于记录第一个获取读锁的线程。也用来提示ThreadLocal变量的性能
  5. 如果一个线程使用了 firstReader，那么它就不需要占用 cachedHoldCounter

注意下 readHolds.get() 的代码，如果ThreadLocal中不存在，会自动加入进去。

下面来看看fullTryAcquireShared方法吧。

源码看到这个，大概就懂了cachedHoldCounter 和 firstReader 的区别。

如果A线程获取读锁后将firstReader指向了自己，在A线程未释放读锁前，B线程获取了读锁，那么B线程不能将firstReader指向自己。但是，A线程将cachedHoldCounter指向自己，当B线程获取到读锁后，可以将修改cachedHoldCounter指向自己。

也就是说如果一个线程在获取到读锁后并使得firstReader指向自己，那么在该线程释放读锁前都可以通过firstReader直接获取重入数，而不需要通过调用readHolds获取。但是cachedHoldCounter在释放读锁前，可能被其他线程修改。

2.2 读锁释放过程

读锁释放过程如下：

  1. 若firstReader指向当前线程，重入数等于1，则将firstReader设置为null；否则重入数减1
  2. 否则，若当前线程的重入数小于等于0，则将当前线程从readHolds中移除；否则重入数减1
  3. 自旋CAS修改state的高16位

在看读锁获取与释放的源码时，发现只有compareAndSetState(c, nextc) 是同步执行的，其他的操作均没有同步，都是顺序执行的。

2.3 写锁获取过程

  1. c != 0 && w == 0: 写锁可用，但是有线程持有读锁(也可能是自己持有)
  2. c != 0 && w !=0 && current != getExclusiveOwnerThread(): 其他线程持有写锁

直接执行执行setState(c + acquires)令我有一些困惑，如果两个写线程，或者一个读线程一个写线程同时执行了上述代码的第25行，不就同时获取写锁了吗？仔细一看才发现，同时只会有一个线程执行到setState(c + acquires)，因为在前面current != getExclusiveOwnerThread()有判断只有独占的线程才能执行！

还有一个疑问，就是当 c == 0时，两个线程同时到了第31行代码，A线程先执行compareAndSetState和setExclusiveOwnerThread，然后B线程再执行compareAndSetState和setExclusiveOwnerThread，这样A线程设置的值不就被B线程覆盖了吗？仔细想了一下，发现问题出在compareAndSetState方法上。

compareAndSetState(int expect, int update)底层是unsafe.compareAndSwapInt(this, stateOffset, expect, update)，其中有两个参数expect，update，expect是原来的值，update是更新之后的值。当A线程执行过compareAndSetState方法后，expect的值已经被改变了，所以B线程执行compareAndSetState肯定不成功，返回false。

另外，还有一点需要注意，写锁获取没涉及firstReader和cachedHoldCounter。
