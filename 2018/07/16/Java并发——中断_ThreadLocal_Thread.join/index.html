【36】Java并发——中断/ThreadLocal/Thread.join

2018-07-16

note：总结在学习Java并发过程中遇到的容易遗忘的知识点

1 线程的中断

想想线程A让B中断的方式有几种：

  1. A线程直接把B线程中断
  2. A线程提醒B你应该中断了，中不中断的由B来决定

1 和 2哪个更合理呢，举个例子来说，高考后填报志愿，1 就是你父母直接帮你填报了；2 就是你父母给你了一些意见，最后填报什么由你自己决定。从这个例子上来看，显然 2 是比较合理的。但是换一个例子，可能就是 1 比较合理了，例如：军官给士兵发送指令。从程序的角度来说，方式 1 可能还会带来其他不必要的麻烦。

最初中断机制，例如 stop()、 suspend()、 resume()，就是实现了方式 1，不过这些方法都已经被弃用，目前主要使用方式 2，那么方式 2 怎么知道自己被人要求中断呢？每个线程都有一个中断标识位，true标识被要求中断，false则不是。方式 2 涉及到的几个方法必须要说说。

  1. public void interrupt() ： 将某个线程的中断标识位设为true
  2. public boolean isInterrupted() ： 线程的中断标识是否是true
  3. public static boolean interrupted() ： 当前线程的中断标识是否是true，获取中断标识后，将中断标识重置成false

InterruptedException

这个异常必须要好好说说，当一个方法中有抛出InterruptedException异常，如BlockingQueue#put、BlockingQueue#take、Object#wait、Thread#sleep等。说明是一个阻塞方法。

阻塞方法不同于一般的要运行较长时间的方法， 阻塞方法的完成还取决于一些外部的事件，例如计时器到期，I/O 完成，或者另一个线程的动作。 阻塞方法可能因为等不到所等的事件而无法终止，因此令阻塞方法可取消 就非常有用。 当一个方法抛出 InterruptedException 时，它是在告诉你，如果执行该方法的线程被中断，它将尝试停止它正在做的事情而提前返回，并通过抛出 InterruptedException 表明它提前返回。 行为良好的阻塞方法应该能对中断作出响应并抛出 InterruptedException，以便能够用于可取消活动中，而不至于影响响应。

当一个方法抛出InterruptedException异常时，说明该线程被其他线程要求中断，并且该线程响应了中断，结束了自己。

sleepThread 和 busyThread 在运行的过程中，被主线程要求中断自己，TimeUnit.SECONDS.sleep(1)是一个阻塞方法，抛出InterruptedException异常，当 sleepThread 被主线程要求中断时，sleepThread 就会抛出异常，并结束自己，当在抛出异常前，Java虚拟机会先将该线程的中断标识位设置为false。输出结果如下：
